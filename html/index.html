CLAUDE.md & Rules
Teaching Claude how YOUR project works
Hands-on 20 min
▼
Three-Level Hierarchy: User (~/.claude/CLAUDE.md) → Project (./CLAUDE.md) → Directory (./src/CLAUDE.md)
What to IncludeTech stack, code style, test commands, architecture decisions, deployment
What NOT to IncludeStandard conventions, API docs, private data — keep under ~200 lines
.claude/rules/Path-specific rules with YAML frontmatter — load only when matching files open
/initAuto-generate CLAUDE.md from your codebase analysis
# Example CLAUDE.md structure
# CLAUDE.md

## Build & Test
npm run build
npm test              # Jest, >80% coverage required
npm run lint          # ESLint + Prettier

## Architecture
Express REST API with PostgreSQL.
Routes in src/routes/, services in src/services/.

## Conventions
- TypeScript strict mode
- Functional components with hooks (React)
- Error responses: { error: string, code: number }

Path-specific Rule Example

# .claude/rules/testing.md
---
paths: "**/*.test.ts"
---
Use describe/it blocks. Prefer jest.spyOn over manual mocks.
Always test error paths. Minimum 3 assertions per test.

Skills & Commands
Reusable workflows and knowledge
Hands-on 30 min
▼
Reference SkillsAuto-loaded knowledge — testing patterns, style guides, domain context
Action SkillsInvoked with /skillname — code review, refactoring workflows
CommandsSimple Markdown shortcuts in .claude/commands/ with $ARGUMENTS
Bundled Skills/batch, /debug, /loop, /simplify, /plan — built-in power tools
Key Difference: Skills have YAML frontmatter with context: fork, tool restrictions, and auto-invocation. Commands are simpler — just Markdown templates.
# .claude/skills/review/SKILL.md
---
name: Code Review
description: Review code for quality and patterns
user-invocable: true
allowed-tools: [Read, Glob, Grep]
---
Review the code at $ARGUMENTS for:
1. Code quality and readability
2. Potential bugs or edge cases
3. Adherence to project conventions

# Simple command: .claude/commands/fix-issue.md
Fix GitHub issue #$ARGUMENTS.
Read the issue, understand the problem,
implement the fix, and run tests.

MCP: External Connections
Connecting Claude to GitHub, Jira, databases & more
Hands-on 35 min
▼
MCPModel Context Protocol — open standard for tool integration
Transport Typesstdio (local processes), HTTP (cloud services)
Scope HierarchyLocal > Project (.mcp.json) > User (~/.claude.json)
Tool DiscoveryAutomatic, lazy-loaded — idle tools use minimal context
# Add an MCP server
claude mcp add --transport http github https://mcp.example.com
claude mcp add --transport stdio mydb -- npx db-mcp-server

# Manage servers
claude mcp list               # Show all configured
claude mcp get github         # Server details
claude mcp remove github      # Remove a server

# In session
/mcp                          # Check status & token costs

# Debug
MCP_DEBUG=1 claude            # Verbose MCP logging

Hooks: Guaranteeing Determinism
Code that ALWAYS runs — independent of the LLM
Hands-on 25 min
▼
Hook TypesCommand hooks (shell scripts), HTTP hooks (POST to URL)
Lifecycle EventsSessionStart, PreToolUse, PostToolUse, Stop, TaskCompleted, …
Decision ControlExit 0 = continue, Exit 2 = block. Stdout goes into Claude context.
Key DistinctionHooks = deterministic (always run). Skills = LLM-driven (may or may not).
Use Cases: Auto-format after every edit, block dangerous commands, run tests on file change, send notifications on task complete.
// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit",
        "hooks": [
          {
            "type": "command",
            "command": "npx prettier --write \"$CLAUDE_FILE_PATH\""
          }
        ]
      }
    ]
  }
}

# Parallel worktrees
claude --worktree -p "add tests for auth" &

Best Practices & Patterns
Learning from experience
Discussion 15 min
▼
When /clearStarting a new unrelated task — fresh context, no noise
When /compactLong session, growing context — compress but keep focus
Context RotAI degrades over time — conflicting instructions, abandoned experiments
Plan FirstAlways review Claude's approach before execution on complex tasks
Prompt HygieneBe specific, not verbose. One task per prompt. Provide examples.
Multi-AgentParallel worktrees for independent tasks, subagents for focused work
