Reusable workflows and automation
Hands-on 30 min
▼
Commands.claude/commands/name.md → /name — simple prompt shortcuts
Skills.claude/skills/name/SKILL.md — complex workflows with frontmatter control
Reference vs. ActionReference (auto-loaded knowledge) vs. Action (invoked with /name)
Frontmatterallowed-tools, context: fork, model, paths, argument-hint
Dynamic Context!`command` injects live shell output. $ARGUMENTS, $1, ${CLAUDE_SKILL_DIR}
Bundled Skills/batch, /debug, /loop, /simplify, /plan — built-in workflows
Rule of thumb: Start with a command. If you need templates, tool restrictions, or isolated context — upgrade to a skill.

07
Subagents
Isolated specialists for focused work
Hands-on 25 min
▼
What Are Subagents?Isolated Claude instances with own context, tools, and role. Returns summary.
Definition.claude/agents/name.md — YAML frontmatter + role prompt
Tool ControlAllowlist (tools: [Read, Grep]) or denylist. Wildcards: Bash(git *), Edit(*.ts)
Built-In AgentsExplore (Haiku, read-only), Plan (specs), General-Purpose (all tools)
InvocationNatural language, @"agent-name", or CLI --agent flag
Key benefit: Tool isolation prevents accidents. Read-only reviewer can't modify code. Each agent gets focused context.
# .claude/agents/code-reviewer.md
---
name: code-reviewer
description: Security and quality reviewer
tools: [Read, Grep, Glob]
model: opus
permissionMode: auto
---

You are an expert code reviewer...

# Invocation patterns
@"code-reviewer" Review auth module
claude --agent code-reviewer
# Claude can also auto-delegate

08
MCP: External Connections
Connecting Claude to the outside world
Hands-on 35 min
▼
What is MCP?Model Context Protocol — open standard for external tool integration
Transport Typesstdio (local processes) or HTTP (cloud services)
Installationclaude mcp add --transport <type> <name> <url/command>
ScopeLocal (project), Project (.mcp.json, team-shared), User (global)
Managementclaude mcp list/get/remove, /mcp in session, MCP_DEBUG=1
Rule of thumb: Local dev → stdio. Cloud/shared → HTTP. Team config → .mcp.json (committed).
# HTTP transport (cloud)
claude mcp add --transport http github \
  https://mcp.example.com/github \
  --header Authorization "Bearer TOKEN"

# stdio transport (local)
claude mcp add --transport stdio filesystem -- \
  npx -y @modelcontextprotocol/server-filesystem /tmp

# Project scope (.mcp.json)
{
  "mcpServers": {
    "github": {
      "transport": "http",
      "url": "https://mcp.github.com",
      "headers": {"Authorization": "Bearer ${GITHUB_TOKEN}"}
    }
  }
}

09
Hooks: Deterministic Guardrails
Code that ALWAYS runs — not optional
Hands-on 15 min
▼
What Are Hooks?Shell scripts at lifecycle events — always execute, 100% deterministic
Key EventsPreToolUse (block), PostToolUse (react), SessionStart (init), Stop (cleanup)
Exit Codes0 = continue (allow), 2 = block tool execution
Hook Input$TOOL_INPUT env var (JSON). Stdout → injected into Claude's context.
Hooks vs. SkillsHooks = guaranteed guardrails. Skills = LLM-driven workflows.
Rule of thumb: If it must happen every time → Hook. If it should happen when useful → Skill.
# .claude/settings.json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "/bin/bash ./scripts/validate-bash.sh"
      }]
    }],
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{
        "type": "command",
        "command": "/bin/bash ./scripts/format-on-save.sh"
      }]
    }]
  }
}

# Exit codes: 0 = allow, 2 = block
# /hooks in session for interactive setup

CLI & Headless Mode
Claude in the pipeline
Hands-on 15 min
▼
Headless Modeclaude -p "prompt" — one-shot execution, no interactive REPL
Output Formats--output-format text|json|stream-json, --json-schema for validation
Tool Control--allowedTools Read,Grep / --disallowedTools Write,Edit,Bash
Permissions in CI--dangerously-skip-permissions + --allowedTools for safe automation
CI/CD IntegrationGitHub Actions PR review with ANTHROPIC_API_KEY secret
Best practice for CI: Combine --dangerously-skip-permissions with --allowedTools to auto-approve only safe operations.
# One-shot execution
claude -p "Analyze test failures" --output-format json

# Read-only, safe for CI
claude -p "Review this code" \
  --allowedTools Read,Grep \
  --dangerously-skip-permissions

# Cost control
claude -p "Full analysis" --max-budget-usd 5

# Custom role
claude -p "Review" --system-prompt "You are a security auditor."

# Session continuity
claude --name "nightly-review" -p "Run scan"

Plugins & Marketplace
Sharing workflows across teams
Demo 10 min
▼
What Is a Plugin?Bundle of skills, commands, hooks, MCP servers — namespaced
Namespacing/plugin-name:skill-name prevents collisions across plugins
Plugin Structureplugin.json manifest + skills/ + hooks/ + .mcp.json
Installationclaude plugin install <name> from marketplace or Git repo
Official Pluginscode-review, commit-commands, hookify, security-guidance, plugin-dev
When to plugin: Team workflows that span repos. When to keep as skill: single-project, experimental.
# Install from marketplace
claude plugin install code-review-toolkit

# Install from Git
claude plugin install git+https://github.com/org/plugin.git

# Management
claude plugin list
claude plugin update code-review-toolkit
claude plugin remove code-review-toolkit

# Namespaced usage
/code-review-toolkit:review src/payment.ts

Claude Code Remote & Web
Browser-based, no local install
Overview 10 min
▼
Claude Code WebBrowser-based at claude.ai — isolated Linux VM on-demand (Max/Enterprise)
Local vs. RemoteRemote for onboarding, quick prototyping. Local for performance and full control.
CoworkDesktop app for non-technical users — VM-based, local files
LimitationsStorage cleared on idle, no Docker/GPU, public URLs only, slower I/O
When Remote vs. Local? Remote for onboarding, pair-programming, quick prototyping, no-setup demos. Local for performance-critical work and full control.
Key Point: Claude Code Web provides the same core experience as the CLI, running in an isolated cloud VM. Great for onboarding and demos.
